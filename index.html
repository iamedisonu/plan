1 Direction

The React frontend should present Isomo through clear programme workspaces, operational screens, and a cited leadership assistant without caring how backend data is stored or organized.

Build one platform

Use a shared shell, consistent navigation, and a single design system.

Keep programme scope clear

Users should always know which programme and cohort they are viewing.

Protect trust

Every number needs a denominator, source, freshness, and warning state when relevant.

One platform Programme workspaces Evidence layer Leadership assistant

2 Key Adjustments

The screen plan is strong. These adjustments make it buildable without coupling the frontend to the current data layout.

Start with Circles

Circles should be the first product workspace. Keep Academy, IRIS, and Graduate in the architecture, but gate each workspace behind its own backend contract.

Dashboard before AI

Build the structured evidence screens first. The assistant becomes more credible when users can open the same metrics, sources, and citations.

Contract-first backend

React consumes stable JSON responses only. The backend may use CSV, Postgres, another store, or a temporary mock layer, but the browser should not know.

Backend validates uploads

The browser can upload and show status, but the backend must validate headers, week labels, promotion status, and logs.

3 Product Shape

The experience should feel like one Isomo product with four workspaces, one leadership assistant, and one shared evidence/audit layer.

Step 1

Choose programme

Start with Circles. Other workspaces can exist as planned states until contracts are ready.

Step 2

Dashboard, schools, indicators, identity, data quality, and evidence.

Step 3

Ask questions

Leadership Q&A uses trusted metrics and returns citations.

Step 4

Act with audit

Uploads, review actions, exports, and sensitive operations are logged.

5 Screen Plan

These screens should stay in the product plan. The first implementation should prioritize shell, dashboard, schools, indicators, data quality, and evidence using mockable API responses.

01

Global Shell

Programme switcher, sidebar navigation, active programme badge, notifications, user menu, assistant entry, and audit/admin links.

Must reset filters, search, selected school, and selected learner on programme switch.
02

Leadership Q&A

Question input, answer card, citation panel, source chips, follow-ups, clarification, retry, regenerate, and cancel.

No trusted answer without source evidence unless backend explicitly marks it safe.
03

Programme Dashboard

Fast snapshot of learners, schools, reporting denominator, coverage, package progress, freshness, and warnings.

Every metric card shows denominator and source/freshness.
04

Identity Ledger

Canonical learners, source IDs, raw IDs, match status, review queue, conflicts, and evidence drawers.

React must not invent matches. Matching decisions come from backend.
05

Manual Uploads

Source selector, week selector, upload, backend validation result, promotion status, and run log.

Backend owns upload handling, validation, promotion, and logging. React only shows the status contract.
06

Tracker Review

Raw submission panel, cleaned submission panel, review queue, void recommendations, closure reason, owner, and status.

Keep system-generated reasons visible, but require human confirmation for sensitive final actions.
07

Indicators Board

Metric cards with value, numerator, denominator, owner, caveat, status, source, freshness, and evidence click-through.

This is the bridge between data and leadership decisions.
08

Learner Journey

Circles learner profile first, then future cross-programme timeline once enrollment contracts exist.

Learner-level views are sensitive and must be role-gated.
09

Data Quality

Backend health, match coverage, learner coverage, stale data warnings, conflicts, and validation results.

This screen should ship early because it protects credibility.
10

Evidence

Metric dictionary, evidence items, generated timestamps, row counts, warnings, and citation previews.

Evidence should open in drawers so users do not lose context.

Frontend boundary

React should only know endpoint names, request shapes, response fields, statuses, and evidence objects. It should not know whether the backend used CSVs, folders, R scripts, Postgres tables, or another internal structure.

8 Assistant Contract

The assistant is not just chat. It is a cited answer surface. It should explain what it knows, where it came from, and what warnings apply. 

{
  "answer_id": "ans_123",
  "thread_id": "thread_123",
  "status": "complete",
  "mode": "structured_metric",
  "answer": "Plain-language answer.",
  "cards": [],
  "tables": [],
  "citations": [],
  "warnings": [],
  "suggested_followups": []
}
{
  "citation_id": "cite_123",
  "source_type": "metric_contract",
  "title": "Learner coverage",
  "source_ref": "evidence:learner_coverage:latest",
  "record_locator": {
    "page": null,
    "section": "coverage_summary",
    "rows": []
  },
  "summary": "Coverage counts from the latest approved backend evidence.",
  "generated_at": "2026-04-15T21:15:38Z"
}

Loading states

thinking, retrieving sources, checking metrics, generating answer, attaching citations, complete.

Answer modes

structured metric, SQL answer, document answer, hybrid answer, cannot answer, needs clarification.

No source case

Show: "No trusted source found." Do not present an uncited answer as reliable.

9 Visual Direction

The UI should feel calm, operational, and trustworthy. Most of the structure should be neutral, with green and blue used intentionally.

Headings and high contrast text.
Muted#54595F
Secondary text and notes.
Border#EDEDED
Lines, separators, quiet structure.

Use skeletons

Dashboard cards, tables, assistant answers, evidence drawers, learner panels, and school details need layout-matching skeletons.

Cache by programme

TanStack Query keys must include programme scope so one workspace never leaks data into another.

Invalidate after actions

Uploads, promotions, review actions, assistant messages, tracker changes, and backend refresh events must refresh affected screens.

10 Rules And Guardrails

These are the rules that keep the frontend aligned with backend contracts instead of backend internals.

Do

  • +Call backend endpoints or mock contract fixtures for trusted results.
  • +Show denominators, freshness, warnings, and evidence.
  • +Keep cache scoped by programme.
  • +Use drawers for evidence, metric definitions, and citations.
  • +Gate sensitive learner and identity views by role.

Do Not

  • -Rebuild backend business logic in React.
  • -Parse files, folders, or pipeline outputs in the browser.
  • -Hardcode counts from current backend files.
  • -Show percentages without denominators.
  • -Show uncited AI answers as trustworthy.

11 First Usable Version

This is the recommended definition of done for the first useful React version.

Area Must Include Why It Matters
Shell Shared shell, programme switcher, role-aware navigation. Keeps one platform feeling while preventing programme confusion.
Circles Workspace Dashboard, school list, school detail, indicators board. Delivers immediate value while the backend can wire real data behind the same contract.
Trust Layer Data-quality page, evidence drawer, freshness badges, evidence badges. Prevents users from trusting numbers blindly.
UX States Skeletons, errors, empty states, retry actions, stable layouts. Makes the product feel reliable under real network conditions.
Frontend Architecture Backend API client, programme-scoped caching, typed response models. Allows backend storage and data organization to change without redesigning screens.

Final sequencing guidance

Build the contract-first interface with mock responses first. Then connect backend data behind the same contracts, add the assistant, and expand to uploads, review workflows, and other programmes.
