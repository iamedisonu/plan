1 Direction

The Vue.js frontend should present Isomo through clear programme workspaces, operational screens, and a standalone cited Isomo Brain workspace without caring how backend data is stored or organized.

Build one platform

Use a shared shell, consistent navigation, and a single design system.

Keep programme scope clear

Users should always know which programme and cohort they are viewing.

Protect trust

Every number needs a denominator, source, freshness, and warning state when relevant.

One platform Programme workspaces Data health Isomo Brain

2 Key Adjustments

The screen plan is strong. These adjustments make it buildable without coupling the frontend to the current data layout.

Start with Circles

Circles should be the first product workspace. Keep Academy, IRIS, and Graduate in the architecture, but gate each workspace behind its own backend contract.

Overview before AI

Build the overview, scholar profiles, schools, and data health surfaces first. Isomo Brain becomes more credible when users can open the same programme context, warnings, and citations.

Contract-first backend

Vue.js consumes stable JSON responses only. The backend may use CSV, Postgres, another store, or a temporary mock layer, but the browser should not know.

Backend validates uploads

The browser can upload and show status, but the backend must validate headers, week labels, promotion status, and logs.

3 Product Shape

The experience should feel like one Isomo product with four workspaces, one standalone Isomo Brain workspace, one global account/settings layer, and one data-health surface per programme.

Step 1

Choose programme

Start with Circles. Other workspaces can exist as planned states until contracts are ready.

Step 2

Inspect programme information

Overview, scholar profiles, schools, and data health for the active workspace.

Step 3

Ask questions

Isomo Brain uses the active workspace context and returns cited answers.

Step 4

Act with audit

Uploads, review actions, exports, and sensitive operations are logged.

6 API Contract Direction

Keep API routes programme-scoped and storage-agnostic. Response contracts should be stable, versioned, and evidence-aware, while backend storage and data organization remain private implementation details.

Phase 1 note

Document these contract targets, but do not connect them in Phase 1. The first deliverable is only the visible Vue.js app skeleton, so endpoints, mock responses, real records, and generated answers should wait until Phase 2.

Backend architecture the Vue app must reflect

  • Sources: Canvas API, Google Drive/Sheets, manual CSV/XLSX, EF SET/DET, Northstar, Typing, and Quill.
  • Foundation: Postgres + pgvector, with Supabase recommended for Auth, RLS, and managed vector support.
  • Brain layer: FastAPI router chooses SQL, RAG, or hybrid paths per question.
  • Model layer: self-hosted model through an llm_client abstraction, with swappable dev/fallback providers.

Frontend space to reserve in Phase 1

  • Workspace-aware data-source badges and data health sections with upload states.
  • Role badge and access-state shells for field staff, leadership, and admin.
  • Isomo Brain routing-state chips: SQL, RAG, hybrid, validating, citing, and saved thread history by workspace.
  • Citation surfaces for SQL run metadata, result rows, document chunks, and generated timestamps.

8 Isomo Brain Workspace

Isomo Brain should be a standalone workspace at /isomo-brain, not a persistent sidebar inside the other product pages. It should feel like a focused question-and-answer workspace with a message thread, saved chat history, composer, answer cards, SQL/RAG/hybrid routing states, citations, and warning states.

Phase 1 Isomo Brain skeleton

Show a full-page Brain layout with a thread history rail or history list shell, an empty message thread, disabled or neutral question composer, routing-state chips, answer card skeleton, citation panel skeleton, and follow-up area shell. Do not show fake answers or fake citations.

Navigation rule

The app may link to Isomo Brain from the main navigation or user menu, but other pages should not reserve a permanent Brain sidebar. Keep page width for the operational screens, and keep chat history scoped to the active workspace.

9 Visual Direction

The UI should feel calm, operational, and trustworthy. Most of the structure should be neutral, with green and blue used intentionally.

Lines, separators, quiet structure.

Use skeletons

Overview cards, scholar lists, school details, data health panels, and Isomo Brain answers need layout-matching skeletons.

Cache by programme

TanStack Query keys must include programme scope so one workspace never leaks data into another.

Invalidate after actions

Uploads, sync refreshes, profile updates, Isomo Brain messages, and backend refresh events must refresh affected screens.

Settings and theme modes

Include a Settings page in Phase 1 with an appearance section for Light, Dark, and System theme modes. The skeleton should prove that the shell, cards, tables, drawers, Isomo Brain page, and forms remain readable in each mode.

10 Rules And Guardrails

These are the rules that keep the frontend aligned with backend contracts instead of backend internals.

Do

  • +In Phase 1, render neutral skeleton screens only; in later phases, call backend endpoints for trusted results.
  • +Show denominators, freshness, warnings, sources, and citations.
  • +Keep cache scoped by programme.
  • +Use drawers for sources, metric definitions, and citations.
  • +Gate sensitive scholar records and profile views by role.

Do Not

  • -Rebuild backend business logic in Vue.js.
  • -Use fake Phase 1 values, names, school records, scholar records, AI answers, or successful upload statuses.
  • -Parse files, folders, or pipeline outputs in the browser.
  • -Hardcode counts from current backend files.
  • -Show percentages without denominators.
  • -Show uncited AI answers as trustworthy.

11 Phase 1 Visual Skeleton Scope

This is the recommended definition of done for Phase 1. The supervisor should be able to open the Vue.js app, move through the pages, and understand how the product will look and behave before any data is connected.

Area Must Include Why It Matters
Shell Shared shell, sidebar, top bar, programme switcher, role badge placeholder, user menu, notification area, responsive page frame. Shows the main structure of the app before content is added.
Page Skeletons Workspace, overview, scholar profiles, schools, data health, Isomo Brain, settings, login, and profile-menu states. Makes every expected page visible and clickable for design review.
Skeleton UI States Overview metrics, scholar search and filters, school tables, data health panels, upload area, Isomo Brain answer cards, drawers, empty states, loading states, and disabled action states. Lets the team judge spacing, hierarchy, flow, and responsive behavior without data.
Project Schema Vue Router route map, page files, feature folders, workspace switcher, profile menu, shared skeleton components, settings page, theme mode control, theme tokens, and CSS structure. Gives developers a clean foundation for later data wiring.
Access And Roles Login page shell, Google SSO button shell, role badge placeholders, access denied shell, and locked sensitive-content placeholders for field staff, leadership, and admin roles. No Data Rule No backend calls, no mock response values, no fake scholar names, no fake school names, no fake Isomo Brain answers, and no fake upload statuses. Keeps Phase 1 focused on visual approval only.

Final sequencing guidance

Build and approve the no-data visual skeleton first. Then connect backend contracts, add real programme data, add Isomo Brain behavior, and expand the same structure to other programmes in later phases.
