1 Direction

The React frontend should present the existing Isomo data system through clear programme workspaces, operational screens, and a cited leadership assistant.

Build one platform

Use a shared shell, consistent navigation, and a single design system.

Keep scope clear

Users should always know which programme and cohort they are viewing.

Protect trust

Every number needs a denominator, source, freshness, and warning state when relevant.

2 Key Adjustments

The screen plan is strong. These adjustments make it buildable against the real Isomo project and reduce delivery risk.

Start with Circles

Circles is the strongest active data lane. Keep Academy, IRIS, and Graduate in the architecture, but do not pretend they are equally ready.

Dashboard before AI

Build the structured evidence screens first. The assistant becomes more credible when users can open the same metrics, sources, and citations.

CSV-backed backend first

Use FastAPI over governed CSV exports first. Move to Postgres and pgvector later without changing the React UI contract.

Backend validates uploads

The browser can upload and show status, but the backend must validate headers, week labels, promotion status, and logs.

3 Product Shape

The experience should feel like one Isomo product with four workspaces, one leadership assistant, and one shared evidence/audit layer.

Step 1

Choose programme

Start with Circles. Other workspaces can exist as planned states until contracts are ready.

Step 2

Inspect data

Dashboard, schools, indicators, identity, data quality, and evidence.

Step 3

Ask questions

Leadership Q&A uses trusted metrics and returns citations.

Step 4

Act with audit

Uploads, review actions, exports, and sensitive operations are logged.

5 Screen Plan

These screens should stay in the product plan. The first implementation should prioritize shell, dashboard, schools, indicators, data quality, and evidence.

01

Global Shell

Programme switcher, sidebar navigation, active programme badge, notifications, user menu, assistant entry, and audit/admin links.

Must reset filters, search, selected school, and selected learner on programme switch.
02

Leadership Q&A

Question input, answer card, citation panel, source chips, follow-ups, clarification, retry, regenerate, and cancel.

No trusted answer without source evidence unless backend explicitly marks it safe.
03

Programme Dashboard

Fast snapshot of learners, schools, reporting denominator, coverage, package progress, freshness, and warnings.

Every metric card shows denominator and source/freshness.
04

Identity Ledger

Canonical learners, source IDs, raw IDs, match status, review queue, conflicts, and evidence drawers.

React must not invent matches. Matching decisions come from backend.
05

Manual Uploads

Source selector, week selector, upload, backend validation result, promotion status, and run log.

Backend owns folder creation, header validation, week validation, promotion, and logging.
06

Tracker Review

Raw submission panel, cleaned submission panel, review queue, void recommendations, closure reason, owner, and status.

Keep system-generated reasons visible, but require human confirmation for sensitive final actions.
07

Indicators Board

Metric cards with value, numerator, denominator, owner, caveat, status, source, freshness, and evidence click-through.

This is the bridge between data and leadership decisions.
08

Learner Journey

Circles learner profile first, then future cross-programme timeline once enrollment contracts exist.

Learner-level views are sensitive and must be role-gated.
09

Data Quality

Pipeline status, match coverage, learner coverage, missing exports, stale exports, conflicts, and validation results.

This screen should ship early because it protects credibility.
10

Evidence

Export manifest, metric dictionary, source table list, generated timestamps, row counts, warnings, and citation previews.

Evidence should open in drawers so users do not lose context.

8 Assistant Contract

The assistant is not just chat. It is a cited answer surface. It should explain what it knows, where it came from, and what warnings apply.

{
  "answer_id": "ans_123",
  "thread_id": "thread_123",
  "status": "complete",
  "mode": "structured_metric",
  "answer": "Plain-language answer.",
  "cards": [],
  "tables": [],
  "citations": [],
  "warnings": [],
  "suggested_followups": []
}
{
  "citation_id": "cite_123",
  "source_type": "csv_export",
  "title": "Learner coverage",
  "path_or_uri": "viz/reports/exports/v2/docs/learner_coverage.csv",
  "record_locator": {
    "rows": [],
    "columns": []
  },
  "summary": "Coverage counts from the latest export.",
  "generated_at": "2026-04-15T21:15:38Z"
}

Loading states

Thinking, retrieving sources, checking metrics, generating answer, attaching citations, complete.

Answer modes

Structured metric, SQL answer, document answer, hybrid answer, cannot answer, needs clarification.

No source case

Show: "No trusted source found." Do not present an uncited answer as reliable.

9 Visual Direction

The UI should feel calm, operational, and trustworthy. Most of the structure should be neutral, with green and blue used intentionally.

Green

#8CC63D — primary actions, success, confirmed records.

Blue

#1DAFEC — links, selected nav, citations, info.

Dark

#303030 — headings and high contrast text.

Muted

#54595F — secondary text and notes.

Border

#EDEDED — lines, separators, quiet structure.

Use skeletons

Dashboard cards, tables, assistant answers, evidence drawers, learner panels, and school details need layout-matching skeletons.

Cache by programme

TanStack Query keys must include programme scope so one workspace never leaks data into another.

Invalidate after actions

Uploads, promotions, review actions, exports, tracker changes, and pipeline reruns must refresh affected screens.

10 Rules and Guardrails

These are the rules that keep the frontend aligned with the Isomo data system.

Do

  • +Call backend endpoints for trusted results.
  • +Show denominators, freshness, warnings, and evidence.
  • +Keep cache scoped by programme.
  • +Use drawers for sources, metric definitions, and citations.
  • +Gate sensitive learner and identity views by role.

Do Not

  • ×Parse raw source files in the browser.
  • ×Hardcode current CSV counts.
  • ×Mix data between programmes.
  • ×Show uncited answers as trustworthy.
  • ×Build broad AI chat before structured metrics exist.

12 Line-by-Line Guidance for the Engineer

This section removes ambiguity. The frontend engineer should read and implement in this order.

1. Build the app shell first.

Create a shared shell with logo, programme switcher, active programme label, notifications, user menu, assistant entry, and evidence/audit links.

2. Make programme state explicit everywhere.

Every screen must know programmeSlug, cohortSlug, userRole, permissions, and dataFreshness. Never allow one programme’s data to appear inside another.

3. Create the Circles dashboard next.

Show canonical learners, canonical schools, denominator, coverage, package progress, freshness, warnings, and source notes.

4. Add school explorer and school detail.

Let users compare schools and inspect one school in detail, including coverage, Canvas activity, grading, and freshness.

5. Add indicators and data quality.

Every indicator card must show a numerator, denominator, source, freshness, owner, caveat, and click-through evidence. Data quality must show missing files, stale exports, and unresolved issues.

6. Add identity and learner views.

Use canonical learner IDs, source-system mappings, and review queues. Never let the browser invent matches.

7. Add operational workflows.

Manual uploads, tracker review, package progress, and Canvas operations must be backend-validated and audit-friendly.

8. Add evidence and citations drawers.

Any number should be openable as evidence. Users should be able to see source files, row counts, metric definitions, warnings, and generated times.

9. Add the assistant last.

The assistant must answer only from trusted sources and display citations. If no source exists, show that clearly instead of guessing.

13 Final Summary

This frontend should make Isomo usable for leadership and operations. It should protect trust by showing where each number came from, how fresh it is, and whether the denominator is complete. It should also create a stable base for assistant features, upload workflows, identity review, and cross-programme reporting later.
